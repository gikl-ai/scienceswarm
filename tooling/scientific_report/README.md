# `scientific_report` — sophisticated HTML reports from a typed spec

A small Python library + CLI that turns a structured JSON description of a
scientific run into a single HTML file. Interactive Plotly figures and
Google Fonts are loaded from public CDNs at view time, so the resulting
page renders without a build step but requires view-time internet access
to look fully styled.

The renderer is theme-driven. Each theme owns its full HTML/CSS/JS surface;
nothing is shared across themes by accident. This is deliberate — adjacent
reports should not look "generated by the same template", and visual
authorship belongs to whoever owns the science being communicated.

Currently shipped themes:

| Theme        | Aesthetic                                   | Used by             |
|---           |---                                          |---                  |
| `broadsheet` | Warm cream policy-journal; navy + signature red + mustard; DM Serif Display + Source Serif 4 + IBM Plex Mono | `scm-ir-quickstart` |

Planned: `editorial` (the dark Fraunces/Newsreader/JetBrains-Mono style used
by the SYK tutorial — to be factored from the inline implementation in
`docs/tutorials/syk-spectral-form-factor/scripts/03_render_report.py` once a
second tutorial wants it).

## Quickstart

```bash
python -m scientific_report --spec path/to/spec.json --out path/to/report.html
```

Optionally override the theme:

```bash
python -m scientific_report --spec spec.json --out report.html --theme broadsheet
```

The renderer requires only the Python standard library; the resulting HTML
loads Plotly, Google Fonts (DM Serif Display, Source Serif 4, Inter Tight,
IBM Plex Mono), and (for some themes) KaTeX from CDNs.

optional unless noted):

## Adding a new theme

Drop a module at `themes/<name>.py` exposing a single function:

```python
def render(spec: dict) -> str:
    ...
```

then register it in `themes/__init__.py` and in the `_THEMES` dict at
`render.py`. Themes are intentionally siloed — copy `broadsheet.py` as a
starting point, but resist factoring shared CSS until at least three themes
exist and the pattern is genuinely common.

## Why a separate skill (and not inline R / inline Python)?

- The visual authorship of a result page is a non-trivial design choice
  ("editorial folio" vs. "broadsheet brief"); pulling it out of the
  pipeline makes those choices reviewable as such.
- Each tutorial's *exporter* (which reads `.rds`/`.npz`/etc. and writes the
  spec) is small and language-native to the tutorial. The renderer is one
  shared Python tool that tutorials in any language can target.
- Once a third tutorial exists in this style, this layer is where shared
  utilities (number formatting, color tokens, accessible contrast checks)
  will accrete.
